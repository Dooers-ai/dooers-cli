# Dooers CLI v2.1 — Simplified Login + Path-Based Routing

**Status:** Implementation doc — ready to build
**Date:** 2026-05-27
**Supersedes parts of:** `docs/superpowers/specs/2026-05-27-dooers-lb-design.md` (URL strategy) and `docs/devops/gcp-lb.md` (LB setup)

---

## 1. Overview

Two changes to the shipped v2 POC:

**Change A — Simpler login.** `dooers auth login --email you@x.com` becomes `dooers login you@x.com`. If the email is omitted, the CLI prompts for it.

**Change B — Path-based routing.** Drop the per-agent subdomain approach (`*.agents.dooers.ai`) in favor of path-based routing under a single domain: `agents.dooers.ai/{agent-identifier}`. This eliminates wildcard DNS and wildcard SSL certificates — both of which are the fiddliest parts of the LB setup.

Both changes are small in code but Change B has one non-obvious technical detail (path-prefix rewrite) that is the crux of the whole approach. It's covered in §3.2.

### Why path-based is simpler on the infra side

| Concern | Subdomain approach (current) | Path-based approach (this doc) |
|---|---|---|
| SSL certificate | Wildcard `*.agents.dooers.ai` — Google-managed wildcard certs need DNS-01 authorization (extra setup, slower) | Single-domain `agents.dooers.ai` — provisions automatically via the load balancer once DNS points at it |
| DNS | Wildcard A record `*.agents.dooers.ai` | One A record `agents.dooers.ai` |
| URL Map structure | One **host rule** per agent | One host rule total; one **path rule** per agent |
| New-agent propagation | New subdomain + cert SAN | New path rule only (faster, no cert involvement) |

The tradeoff path-based historically carried — "the agent's own code must be path-aware" — is **eliminated by URL path-prefix rewrite at the load balancer** (§3.2). The agent keeps serving from `/`; the LB strips the `/{agent-id}` prefix before forwarding.

## 2. Change A — Simplified login

### 2.1 New command surface

| Before | After |
|---|---|
| `dooers auth login --email you@x.com` | `dooers login you@x.com` |
| `dooers auth login` (prompts for email) | `dooers login` (prompts for email) |
| `dooers auth whoami` | `dooers whoami` *(promoted; see note)* |
| `dooers auth logout` | `dooers logout` *(promoted; see note)* |

**Decision:** promote all three auth commands to top-level. `login` is the one the user explicitly asked to simplify; promoting `whoami`/`logout` alongside it keeps the surface consistent (no reason `login` is top-level but its siblings are nested). The `auth` subcommand group is removed.

If you'd rather keep `whoami`/`logout` under `auth` and only promote `login`, that's a one-line change — but the symmetric option is cleaner and is what this doc specifies.

```python
app.command(name="whoami", help="Show the currently authenticated user.")(auth.whoami)
app.command(name="logout", help="Clear local credentials.")(auth.logout)
```

## 3. Change B — Path-based routing

### 3.1 New URL convention

```
prod:      https://agents.dooers.ai/{agent-id-safe}
non-prod:  https://agents.dooers.ai/{agent-id-safe}-{env}
```

Examples:
- `https://agents.dooers.ai/ag-7q4r` (prod)
- `https://agents.dooers.ai/ag-7q4r-dev` (dev)
- `https://agents.dooers.ai/ag-7q4r-stg` (staging)

The path segment is exactly the string that used to be the subdomain label — `safe_agent_id(agent_id)` (underscores → hyphens, lowercased), with the env suffix on non-prod. **Same logic; it just moves from before the domain to after it.** `agents.dooers.ai` is a single fixed host.

### 3.2 The crux: path-prefix rewrite (so agent code stays unchanged)

A naïve path-based LB would forward `agents.dooers.ai/ag-7q4r/health` to the agent's Cloud Run service as `/ag-7q4r/health`. The agent serves `/health`, so it would 404 — and every root-relative link in the agent's HTML/JSON would break. That's the classic objection to path-based routing.

**We avoid it entirely with a URL-rewrite on the path rule.** GCP external HTTPS load balancers support `pathPrefixRewrite`: the LB strips the matched prefix before forwarding to the backend.

- Request: `GET agents.dooers.ai/ag-7q4r/health`
- Path rule matches `/ag-7q4r/*`
- `pathPrefixRewrite = "/"` rewrites the matched prefix `/ag-7q4r` → `/`
- Backend (Cloud Run) receives: `GET /health`

The agent keeps serving from `/`. It never knows it's behind a path prefix. This is the single most important detail in this doc — without it, path-based routing is a bad idea; with it, it's strictly simpler than subdomains.

### 3.3 URL Map structure change

**Before (subdomain):** one host rule per agent, each pointing at a one-rule path matcher.

```
URL Map
├── host_rule: ag-7q4r-dev.agents.dooers.ai → path_matcher "agent-ag-7q4r-dev-pm"
├── host_rule: ag-8h2k.agents.dooers.ai     → path_matcher "agent-ag-8h2k-pm"
└── default_service: 404-bs
```

**After (path-based):** one host rule total; one shared path matcher with many path rules.

```
URL Map
├── host_rule: agents.dooers.ai → path_matcher "agents-pm"
├── path_matcher "agents-pm"
│   ├── default_service: 404-bs
│   ├── path_rule: ["/ag-7q4r", "/ag-7q4r/*"]      → bs-ag-7q4r       (rewrite → /)
│   ├── path_rule: ["/ag-7q4r-dev", "/ag-7q4r-dev/*"] → bs-ag-7q4r-dev (rewrite → /)
│   └── path_rule: ["/ag-8h2k", "/ag-8h2k/*"]      → bs-ag-8h2k       (rewrite → /)
└── default_service: 404-bs
```

The Serverless NEG and Backend Service per agent are **unchanged** — same create-or-get, same names (`agent-{safe}-{env}-neg`, `agent-{safe}-{env}-bs`). Only the URL Map mutation changes: instead of adding a host rule + dedicated path matcher, we add a path rule to the single shared `agents-pm` matcher.

### 3.4 Code changes — `loadbalancer.py`

**Naming helpers** (`packages/dooers-push/src/dooers_push/gcp/loadbalancer.py`):

- `safe_agent_id` — unchanged.
- Replace `host_for(agent_id, env, lb_domain)` with `path_segment_for(agent_id, env)` returning just the segment (no domain):

```python
def path_segment_for(agent_id: str, env: str) -> str:
    """Return the per-agent path segment (no leading slash).

    Prod drops the env suffix; non-prod keeps it.
    path_segment_for('ag_7q4r', 'prod') -> 'ag-7q4r'
    path_segment_for('ag_7q4r', 'dev')  -> 'ag-7q4r-dev'
    """
    safe = safe_agent_id(agent_id)
    return safe if env == "prod" else f"{safe}-{env}"
```

- `neg_name`, `bs_name` — unchanged.
- Remove `path_matcher_name(agent_id, env)` (no longer per-agent). Add a module constant for the shared matcher name:

```python
SHARED_PATH_MATCHER = "agents-pm"
```

**`register_agent`** — the orchestration is the same shape; only the URL Map step and the returned URL change:

```python
async def register_agent(self, agent_id: str, env: str) -> str:
    seg = path_segment_for(agent_id, env)
    neg_url = await self._ensure_neg(agent_id, env)
    bs_url = await self._ensure_backend_service(agent_id, env, neg_url)
    await self._upsert_path_rule(seg, bs_self_link=bs_url)
    return f"https://{self.domain}/{seg}"
```

**`_upsert_path_rule`** — replaces `_update_url_map`. GET the URL Map, find the shared `agents-pm` path matcher, find the path rule whose `paths` includes `/{seg}` (idempotent replace) or append a new one, then PATCH:

```python
async def _upsert_path_rule(self, seg: str, *, bs_self_link: str) -> None:
    client = compute_v1.UrlMapsClient()
    loop = asyncio.get_running_loop()
    paths = [f"/{seg}", f"/{seg}/*"]

    def _get_and_patch() -> None:
        try:
            url_map = client.get(project=self.project_id, url_map=self.url_map_name)
        except gcp_exceptions.NotFound as e:
            raise LBError(
                f"URL map {self.url_map_name!r} not found — has devops completed setup?",
                operation="url_map_get", cause=e,
            ) from e

        # Locate the shared path matcher.
        pm = next((m for m in url_map.path_matchers if m.name == SHARED_PATH_MATCHER), None)
        if pm is None:
            raise LBError(
                f"path matcher {SHARED_PATH_MATCHER!r} missing — re-run gcp setup §LB",
                operation="url_map_pm_missing",
            )

        new_rule = compute_v1.PathRule(
            paths=paths,
            service=bs_self_link,
            route_action=compute_v1.HttpRouteAction(
                url_rewrite=compute_v1.UrlRewrite(path_prefix_rewrite="/")
            ),
        )

        # Idempotent: replace an existing rule that owns /{seg}, else append.
        existing_idx = next(
            (i for i, r in enumerate(pm.path_rules) if f"/{seg}" in r.paths), None
        )
        if existing_idx is not None:
            pm.path_rules[existing_idx] = new_rule
        else:
            pm.path_rules.append(new_rule)

        op = client.patch(
            project=self.project_id, url_map=self.url_map_name, url_map_resource=url_map
        )
        op.result(timeout=120)

    try:
        await loop.run_in_executor(None, _get_and_patch)
        logger.info("lb_op=upsert_path_rule seg=%s ok", seg)
    except LBError:
        raise
    except gcp_exceptions.PermissionDenied as e:
        raise LBError(f"permission denied patching URL Map: {e}",
                      operation="url_map_patch", cause=e) from e
    except gcp_exceptions.GoogleAPIError as e:
        raise LBError(f"failed to patch URL Map: {e}",
                      operation="url_map_patch", cause=e) from e
```

**`unregister_agent`** — the teardown order is the same (URL rule → BS → NEG), but `_remove_url_map_host_rule` becomes `_remove_path_rule`: GET the URL Map, drop the path rule whose `paths` includes `/{seg}` from the shared `agents-pm` matcher, PATCH. The BS and NEG delete helpers are unchanged.

**`wait_until_reachable`** — unchanged (polls whatever URL it's handed; now a path URL).

### 3.5 Settings

`packages/dooers-push/src/dooers_push/settings.py` — `lb_domain` default stays `agents.dooers.ai` (now a single host, not a wildcard parent). `lb_url_map` stays `dooers-agents-url-map`. No new settings needed. The `lb_region` stays.

### 3.6 `dooers-protocol` / CLI / agent record

**No changes needed.** `PushResponse.url`, `AgentRecord.deployed_url`, and the CLI's "Live at: {url}" print all carry whatever string the server produces. The string is now a path URL instead of a subdomain URL — same field, different value. This is the same reason the original LB integration needed no protocol change.

### 3.7 Tests to update

- `packages/dooers-push/tests/test_lb_naming.py` — replace `host_for` tests with `path_segment_for` tests:
  - `path_segment_for("ag_7q4r", "prod") == "ag-7q4r"`
  - `path_segment_for("ag_7q4r", "dev") == "ag-7q4r-dev"`
- `packages/dooers-push/tests/test_lb_manager.py` — replace the `_update_url_map` tests with `_upsert_path_rule` tests:
  - appends a path rule with `paths == ["/ag-7q4r-dev", "/ag-7q4r-dev/*"]` and `route_action.url_rewrite.path_prefix_rewrite == "/"`
  - idempotent: re-registering the same seg replaces in place (one rule, not two)
  - raises `LBError` if URL map / shared matcher missing
  - `register_agent` returns `https://agents.dooers.ai/ag-7q4r-dev`
  - prod returns `https://agents.dooers.ai/ag-7q4r`

### 3.8 Limits note

A single URL Map supports a large but finite number of path rules (historically ~10,000 across all path matchers; verify current quota at build time). The sharding strategy is the same as before: when one URL Map fills, stand up a second LB and route new agents there. For the POC scale this is a non-issue.

## 4. DevOps step-by-step (GCP) — your part

This replaces the LB section of `docs/devops/gcp-lb.md`. Everything is one-time except where noted. Substitute `<PROJECT_ID>` and assume region `us-central1` throughout. You can do all of this in the console; `gcloud` equivalents are given because they're faster and copy-pasteable.

### Prerequisites

- `gcloud` installed and authenticated (`gcloud auth login`, `gcloud config set project <PROJECT_ID>`). Install steps: see `docs/devops/gcp-lb.md` §"Installing gcloud on Ubuntu".
- Owner or (Compute Admin + Cloud Run Admin + DNS Admin + Project IAM Admin) on the project.
- `agents.dooers.ai` DNS managed somewhere you can edit (Cloud DNS assumed below).

### 4.1 Service account for dooers-push

The `dooers-push` Cloud Run service runs as this identity and needs permission to drive Cloud Build, Cloud Run, GCS, Artifact Registry, and the load balancer.

```bash
# Create (skip if it already exists from v1 deploy-service)
gcloud iam service-accounts create agent-deploy-service \
  --display-name="Dooers push pipeline" \
  --project=<PROJECT_ID>

SA="agent-deploy-service@<PROJECT_ID>.iam.gserviceaccount.com"

# Grant the roles dooers-push needs
for ROLE in \
  roles/cloudbuild.builds.editor \
  roles/run.admin \
  roles/storage.admin \
  roles/artifactregistry.writer \
  roles/compute.loadBalancerAdmin \
  roles/iam.serviceAccountUser ; do
  gcloud projects add-iam-policy-binding <PROJECT_ID> \
    --member="serviceAccount:$SA" --role="$ROLE"
done
```

`roles/iam.serviceAccountUser` is needed because Cloud Build deploys Cloud Run services *as* this service account (act-as). `roles/compute.loadBalancerAdmin` is what lets dooers-push edit the URL Map per push.

### 4.2 Deploy dooers-push to Cloud Run

```bash
REGION=us-central1
REPO=agents   # Artifact Registry repo; create if missing:
gcloud artifacts repositories create $REPO \
  --repository-format=docker --location=$REGION --project=<PROJECT_ID> 2>/dev/null || true

# Build + push the image (run from packages/dooers-push/)
cd packages/dooers-push
gcloud builds submit --tag $REGION-docker.pkg.dev/<PROJECT_ID>/$REPO/dooers-push:dev \
  --project=<PROJECT_ID> .

# Deploy
gcloud run deploy dooers-push-dev \
  --image=$REGION-docker.pkg.dev/<PROJECT_ID>/$REPO/dooers-push:dev \
  --region=$REGION --platform=managed --allow-unauthenticated \
  --service-account=agent-deploy-service@<PROJECT_ID>.iam.gserviceaccount.com \
  --set-env-vars="GCP_PROJECT_ID=<PROJECT_ID>,GCP_REGION=$REGION,BUCKET_NAME=dooers-agents-deploy,ARTIFACT_REPO=$REPO,CORE_API_URL=https://api.dev.dooers.ai,ENVIRONMENT=dev,DOOERS_LB_DOMAIN=agents.dooers.ai,DOOERS_LB_URL_MAP=dooers-agents-url-map" \
  --cpu=1 --memory=1Gi --timeout=600 \
  --project=<PROJECT_ID>
```

Notes:
- Create the GCS bucket first if it doesn't exist: `gcloud storage buckets create gs://dooers-agents-deploy --location=$REGION --project=<PROJECT_ID>`.
- Cloud Run gives the service its own `*.run.app` URL. That's the **control-plane** URL the CLI's `--push-url` / `DOOERS_PUSH_URL` points at. It is unrelated to the agent data-plane URLs (`agents.dooers.ai/...`).
- Capture the printed URL; you'll set `DOOERS_PUSH_URL` to it when running the CLI.
- The startup hook validates env vars — if any required var is missing, the service fails to start (by design). Check logs if the revision doesn't go healthy.

Verify:
```bash
curl https://dooers-push-dev-xxxxx.a.run.app/health   # → {"status":"ok"}
```

### 4.3 Load balancer (path-based)

**Step 1 — Reserve a global static IPv4**
```bash
gcloud compute addresses create dooers-agents-lb-ip --global --project=<PROJECT_ID>
gcloud compute addresses describe dooers-agents-lb-ip --global \
  --format='value(address)' --project=<PROJECT_ID>     # note this IP
```

**Step 2 — Single-domain managed SSL cert** (no wildcard — this is the simplification)
```bash
gcloud compute ssl-certificates create dooers-agents-cert \
  --domains=agents.dooers.ai --global --project=<PROJECT_ID>
```

**Step 3 — Placeholder 404 backend** (default for unmatched paths)
```bash
# Tiny Cloud Run service that returns 404/hello for unknown paths
gcloud run deploy dooers-agents-default-404 \
  --image=gcr.io/cloudrun/hello --region=$REGION \
  --allow-unauthenticated --cpu-throttling --project=<PROJECT_ID>

gcloud compute network-endpoint-groups create dooers-agents-default-neg \
  --region=$REGION --network-endpoint-type=serverless \
  --cloud-run-service=dooers-agents-default-404 --project=<PROJECT_ID>

gcloud compute backend-services create dooers-agents-default-404-bs \
  --global --protocol=HTTPS --project=<PROJECT_ID>
gcloud compute backend-services add-backend dooers-agents-default-404-bs \
  --global --network-endpoint-group=dooers-agents-default-neg \
  --network-endpoint-group-region=$REGION --project=<PROJECT_ID>
```

**Step 4 — URL Map with a named path matcher** (this is the key structural difference from the subdomain setup)
```bash
# Create the URL map with the 404 default
gcloud compute url-maps create dooers-agents-url-map \
  --default-service=dooers-agents-default-404-bs --project=<PROJECT_ID>

# Add a host rule for agents.dooers.ai pointing at a named path matcher "agents-pm".
# dooers-push appends per-agent path rules to this matcher on every push.
gcloud compute url-maps add-path-matcher dooers-agents-url-map \
  --path-matcher-name=agents-pm \
  --default-service=dooers-agents-default-404-bs \
  --new-hosts=agents.dooers.ai \
  --project=<PROJECT_ID>
```

After this, the URL Map has: host `agents.dooers.ai` → path matcher `agents-pm` (empty path rules, 404 default). dooers-push fills in the path rules.

**Step 5 — Target HTTPS proxy + global forwarding rule**
```bash
gcloud compute target-https-proxies create dooers-agents-https-proxy \
  --url-map=dooers-agents-url-map \
  --ssl-certificates=dooers-agents-cert --project=<PROJECT_ID>

gcloud compute forwarding-rules create dooers-agents-https-rule \
  --address=dooers-agents-lb-ip --target-https-proxy=dooers-agents-https-proxy \
  --global --ports=443 --project=<PROJECT_ID>
```

**Step 6 — DNS (single A record, no wildcard)**
```bash
# Replace 34.120.x.x with the IP from Step 1. Cloud DNS zone "dooers-ai" assumed.
gcloud dns record-sets create agents.dooers.ai. \
  --zone=dooers-ai --type=A --ttl=300 --rrdatas=34.120.x.x --project=<PROJECT_ID>
```
If DNS is elsewhere (Cloudflare/Route53), add an `A` record `agents.dooers.ai` → the static IP, TTL ~300.

**Step 7 — Wait for the cert to go ACTIVE**
```bash
gcloud compute ssl-certificates describe dooers-agents-cert --global \
  --format='value(managed.status)' --project=<PROJECT_ID>     # PROVISIONING → ACTIVE
```
Single-domain managed certs provision automatically once DNS resolves to the LB IP and the LB is serving on 443. Typically 15–60 min. Do not proceed to the demo until `ACTIVE`.

### 4.4 Verify the whole chain


| URL Map | `dooers-agents-url-map` | you, once |
| Shared path matcher | `agents-pm` | you, once (Step 4) |

## 5. What changes vs. the previously-shipped LB integration

| Aspect | Shipped (subdomain) | This doc (path-based) |
|---|---|---|
| `host_for()` | returns `{seg}.{domain}` | replaced by `path_segment_for()` returning `{seg}` |
| `_update_url_map` | adds **host rule** + per-agent path matcher | `_upsert_path_rule` adds **path rule** to shared `agents-pm` |
| `path_matcher_name()` | per-agent | removed; `SHARED_PATH_MATCHER = "agents-pm"` constant |
| Returned URL | `https://{seg}.agents.dooers.ai` | `https://agents.dooers.ai/{seg}` |
| Cert | wildcard `*.agents.dooers.ai` | single `agents.dooers.ai` |
| DNS | wildcard A record | single A record |
| Path-prefix rewrite | n/a | **required** (`UrlRewrite.path_prefix_rewrite="/"`) |
| NEG / Backend Service | per-agent | per-agent (unchanged) |
| Idempotency | replace host rule | replace path rule (same pattern) |

`gcp-lb.md` should be updated to match §4 of this doc (single-domain cert, `add-path-matcher` step, single A record). The subdomain version can be kept as an appendix or deleted.

## 6. Estimated effort

- **Change A (login):** ~1 hour (cli.py registration + auth.py argument + smoke test + 3 doc updates).
- **Change B (path routing):** ~3-4 hours (loadbalancer.py `path_segment_for` + `_upsert_path_rule` + `_remove_path_rule` + test rewrites + verify the `PathRule.route_action` shape against the live compute API).
- **DevOps (§4):** ~1 hour of commands + 15-60 min cert wait.

The riskiest unknown is confirming GCP accepts a `PathRule` with both `service` and `route_action.url_rewrite` on an external HTTPS LB URL map. Validate that early with a throwaway `gcloud compute url-maps import` or a single SDK call before building the full `_upsert_path_rule`. If GCP rejects the combination, the fallback is `route_rules` (the newer matcher syntax) instead of `path_rules` — same concept, slightly different field names.
